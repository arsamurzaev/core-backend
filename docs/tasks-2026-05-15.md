# Задачи на 15 мая 2026

## Статус

Выполнено 17 мая 2026.

- Складские изменения из cart/order flow теперь выносятся из Prisma transaction как отложенные domain events.
- Публикация событий происходит после успешного commit через существующий post-transaction invalidate hook.
- `reserveCartStockTx`, `consumeCompletedOrderStockTx`, `releaseCartReservationsTx` и expired reservation flow теперь передают точные `previousStock`/`nextStock`.
- Прямой cache bump сохранен как fallback, если event bus недоступен или событий нет.
- Проверки пройдены: `npm.cmd test -- --runInBand --no-cache inventory cart`, `npm.cmd test -- --runInBand --no-cache`, `npm.cmd run build`.

## Контекст

- Backend уже переведен дальше в сторону domain events.
- `variant.stock_changed` и `variant.price_changed` теперь несут `previous/next`, `source`, `reason`, `integrationId`, `externalId`.
- MoySklad stock apply, MoySklad variant sync и manual inventory adjustment уже публикуют точные события.
- Проверки на конец дня: `npm.cmd test -- --runInBand --no-cache` и `npm.cmd run build` проходят.

## Главная задача

Довести складские события из корзины и заказа до надежной модели after-commit/outbox.

## План работ

1. Спроектировать минимальный внутренний after-commit/outbox механизм для модульного монолита.
   - Не публиковать события внутри незавершенной Prisma transaction.
   - Сохранять события или возвращать их из transaction наружу и dispatch-ить только после успешного commit.
   - Сразу заложить идемпотентность по `eventId`/business key.

2. Расширить inventory repository/service для точных stock changes в cart/order flow.
   - `reserveCartStockTx`: собирать изменения по `variantId`.
   - `releaseCartReservationsTx`: собирать изменения по `variantId`.
   - `consumeCompletedOrderStockTx`: собирать изменения по `variantId`.
   - Для каждого изменения иметь `previousStock`, `nextStock`, `source`: `cart`, `order` или `system`.

3. Подключить публикацию событий после commit.
   - Cart reservation: `variant.stock_changed` с reason `cart_reservation`.
   - Cart release: `variant.stock_changed` с reason `cart_reservation_release`.
   - Order consume: `variant.stock_changed` с reason `completed_order_stock_consume`.
   - Expired reservation cron: `variant.stock_changed` с reason `expired_reservation_release`.

4. Убрать дублирующую прямую инвалидацию кеша там, где событие уже публикуется.
   - Сохранить fallback на прямой bump, если event bus недоступен.
   - Проверить, что `PRODUCTS_CACHE_VERSION` и `CATEGORY_PRODUCTS_CACHE_VERSION` bump-аются один раз.

5. Добавить тесты.
   - Unit: события не dispatch-ятся до успешного transaction commit.
   - Unit: reserve/release/consume публикуют корректные previous/next.
   - Unit: rollback/throw не публикует события.
   - Regression: текущие cart/order/inventory тесты остаются зелеными.

6. Прогнать проверки.
   - `npm.cmd test -- --runInBand --no-cache inventory cart`
   - `npm.cmd test -- --runInBand --no-cache`
   - `npm.cmd run build`

## Следующий шаг: durable outbox

Выполнено 17 мая 2026.

- Добавлена Prisma-модель `DomainEventOutbox` и enum `DomainEventOutboxStatus`.
- `DOMAIN_EVENT_DISPATCHER` теперь пишет события в outbox перед доставкой в in-process bus.
- Добавлен cron-drain `domain-event-outbox-drain` для повторной обработки `PENDING`/`FAILED` событий.
- Для cart/order inventory flow события `variant.stock_changed` сохраняются в outbox внутри текущей Prisma transaction.
- `InProcessDomainEventBus` больше не помечает упавшее событие как обработанное, поэтому retry возможен.
- Добавлены тесты на persist-before-dispatch, failed dispatch, drain pending, retry after handler failure и transaction-aware inventory outbox append.

Проверка пройдена:

- `npm.cmd test -- --runInBand --no-cache domain-events inventory`
- `npm.cmd run build`
- `npm.cmd test -- --runInBand --no-cache`

## Следующий шаг: outbox diagnostics

Выполнено 17 мая 2026.

- Добавлен `DomainEventOutboxDiagnosticsService` для production-диагностики outbox.
- Добавлены admin endpoints:
  - `GET /admin/domain-events/outbox` — список событий с фильтрами `status`, `catalogId`, `eventType`, `aggregateType`, `aggregateId`, `limit`;
  - `GET /admin/domain-events/outbox/stats` — счетчики по статусам и быстрые индикаторы pending/failed;
  - `POST /admin/domain-events/outbox/:id/retry` — retry одного pending/failed события;
  - `POST /admin/domain-events/outbox/retry-failed` — retry failed событий по `catalogId`/`eventType`;
  - `POST /admin/domain-events/outbox/drain` — ручной запуск drain.
- `DomainEventOutboxDispatcher` получил `retryByIds`, чтобы ручной retry использовал тот же delivery path, что и cron-drain.
- Добавлены DTO для Swagger и валидации admin-запросов.
- Добавлены тесты на list/stats/retry/drain diagnostics и admin delegation.

Проверка пройдена:

## Следующий шаг: outbox retention cleanup

Выполнено 17 мая 2026.

- Добавлена cleanup-политика для `DomainEventOutbox`: удаляются только старые `PROCESSED` события.
- `FAILED` события не удаляются автоматически и остаются для ручной диагностики/retry.
- Добавлены env-настройки:
  - `DOMAIN_EVENT_OUTBOX_PROCESSED_RETENTION_DAYS`, по умолчанию `30`;
  - `DOMAIN_EVENT_OUTBOX_CLEANUP_CRON`, по умолчанию `0 4 * * *`;
  - `DOMAIN_EVENT_OUTBOX_CLEANUP_TIMEZONE`, по умолчанию `Europe/Moscow`;
  - `DOMAIN_EVENT_OUTBOX_CLEANUP_LIMIT`, по умолчанию `5000`.
- Добавлен cron `domain-event-outbox-cleanup`.
- Добавлен admin endpoint `POST /admin/domain-events/outbox/cleanup` для ручной очистки processed-событий.
- Stats outbox расширены:
  - `processedRetentionDays`;
  - `processedRetentionCutoff`;
  - `processedOlderThanRetention`;
  - `failedOlderThan1Day`;
  - `failedOlderThan7Days`;
  - `failedOlderThan30Days`.
- Cleanup идет пачкой через предварительный выбор `id`, чтобы не пытаться удалить бесконечный объем за один проход.

Проверка пройдена:

- `npm.cmd test -- --runInBand --no-cache domain-events admin`
- `npm.cmd run build`
